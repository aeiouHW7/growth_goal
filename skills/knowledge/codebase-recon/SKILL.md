# 代码库侦察

> 先查再写。不了解现有实现就动手 = 猜测。代码是事实，文档可能过时。能复用就复用，不另起炉灶。

**分层结构**：
```
通用侦察（所有场景前执行，只执行一次）
    ↓
各阶段专项侦察（叠加执行）
    ├── explore — 现有代码边界
    ├── propose — 实现细节、表结构、可复用组件
    ├── apply   — 字段命名、代码风格、工具类
    └── verify  — 测试模式、fixture、断言方式
```

**查找工具**：
- 查代码：`grep` / `search_symbols` / `find_references`
- 查数据库：检查 `backend/prisma/schema.prisma` 或 `init SQL` 文件

---

## 代码定位策略

找不到相关实现时，按以下顺序递进：

1. **直接 grep**：用业务术语、中文名、英文名多试几个关键词
2. **读 `10_DOCS/business/`**：从业务文档找到正确的技术术语或模块名，再回去 grep
3. **读 `openspec/specs/`**：历史规格记录往往包含相关模块的改动，顺着找到代码位置
4. **读目录结构**：看 `backend/src/` 或 `frontend/src/` 的目录结构，从模块名推断划分
5. **顺链路读**：找到入口后，用 `goto_definition` / `find_references` 从 Controller → Service → Repository 完整走一遍

## 通用侦察（所有场景前执行，只执行一次）

目标：建立对业务域和现有系统的基础认知，为后续专项侦察和提问提供事实依据。

**文档轨**（了解意图和约束）：

- [ ] **业务规则**：查 `10_DOCS/business/` 相关文档，了解业务背景、流程规则、历史决策
- [ ] **技术约束**：查 `10_DOCS/technical/` 相关文档，了解架构决策、技术选型、已知限制
- [ ] **历史方案**：查 `openspec/changes/` 有没有相关的历史 proposal/design，了解过去怎么决策的

**代码轨**（了解现实）：

- [ ] **找到核心入口**：用业务关键词 grep，定位涉及的核心 Service/Controller/页面
- [ ] **读主干流程**：读核心功能从入口到出口的主干逻辑（不是细节，是"经过哪些环节、数据怎么流转"）
- [ ] **识别上下游**：这个模块调了谁、谁调了它，依赖关系是什么

**双轨对比**：

完成后主动对比文档和代码是否一致。**文档说应该这样，但代码实际是那样**的偏差，往往是最值得追问的点。

## explore 阶段：背景调研

目标：聊需求前先了解现有实现，避免讨论出脱离现实的方案。

- [ ] **知识库检索**：查 `10_DOCS/business/` 相关业务文档，了解业务背景和已有规则
- [ ] **找到相关模块的现有代码**：按上方"代码定位策略"找到涉及的 Service/Controller/组件
- [ ] **了解现有数据模型**：检查 `backend/prisma/schema.prisma` 或数据库 DDL 文件，了解现有字段和关联关系
- [ ] **识别约束和技术债**：现有实现中有哪些限制会影响新需求的方案方向

---

## propose 阶段：方案调研

目标：确保提案方案与现有实现一致，不重复造轮子。

### 1. 需求输入解析

- [ ] 读取需求输入（explore 阶段的结论 / 90_PLANNING 需求文档 / 用户描述）
- [ ] 提取本次提案的范围边界（只做哪个模块/哪些功能点）
- [ ] 识别涉及的业务实体（表、接口、页面）
- [ ] **内容完整性检查（不可跳过）**：
  - 发现缺失时主动询问用户补充，补充内容固化到 proposal 的 `## 补充资料` 章节（见下方"补充资料规范"）

### 2. 现有实现调研（最关键，不可跳过）

#### 2.1 后端

- [ ] **找到相似功能的 Service/Controller**：按上方"代码定位策略"找到最接近的现有实现
- [ ] **读完整实现**：要读核心方法的逻辑（入参、出参、调用链、异常处理），不是看文件名
- [ ] **理解现有链路**：从 Controller → Service → Repository/Repository → 数据库，完整走一遍
- [ ] **确认可复用的工具类/枚举/常量**：grep 关键词，看 utils/、enums/、constants/ 下有没有现成的

#### 2.2 前端

- [ ] **找到目标页面的现有代码**：确认页面路径、路由配置、columns 定义、API 调用
- [ ] **读页面的交互模式**：现有页面怎么做多选、怎么做批量操作、怎么调接口
- [ ] **确认组件库用法**：table component、search form component 等组件的 props 和 slot 用法
- [ ] **确认已有的公共 hook/工具函数**：避免重复封装

#### 2.3 数据库

- [ ] **查相关表的完整 DDL**：用 `database schema files` 查实际数据库表结构（字段名、类型、索引），比翻 SQL 文件更准确
- [ ] **确认字段是否已存在**：新需求要用的字段在表里有没有，类型是否匹配
- [ ] **查关联表关系**：主子表的外键关联、业务单号的关联方式

### 3. 技术方案验证

- [ ] **打印/导出/上传等特殊能力**：确认现有实现方式（SDK？URL？第三方服务？），不要假设
- [ ] **权限模型**：确认目标页面的权限 key、按钮权限配置方式
- [ ] **国际化**：确认 i18n key 的命名规范和已有的 key

### 4. 知识库检索

- [ ] 查 `10_DOCS/business/` 相关业务文档
- [ ] 查 `10_DOCS/technical/` 相关技术文档（如 database-standards.md）
- [ ] 查 `10_DOCS/api/` 相关接口文档

### 调研输出

调研完成后，在脑中形成以下结论，再开始写 proposal：

1. **现有链路是什么**：从用户操作到数据库的完整路径
2. **本次变更在哪里切入**：新增接口？扩展现有接口？新增页面？修改现有页面？
3. **能复用什么**：现有的 Service、工具类、组件、hook
4. **需要新建什么**：哪些是现有代码覆盖不到的
5. **风险点**：现有实现中的限制、技术债、兼容性问题

## apply 阶段：编码调研

目标：写代码前确认要遵循的模式，不偏离现有风格。

- [ ] 读取 design.md 和 tasks.md，明确当前任务的技术方案
- [ ] 按上方"代码定位策略"找到最相似的现有实现，读完整代码（不是看文件名）
- [ ] 确认要复用的工具类、枚举、常量、组件、hook
- [ ] 用 `database schema files` 确认数据库表 DDL（字段名、类型要精确匹配）
- [ ] 确认国际化 key 命名规范和已有 key
- [ ] 确认权限 key 和按钮权限配置方式

## verify 阶段：测试调研

目标：写测试前确认现有测试模式，复用 fixture 和断言方式。

- [ ] 按上方"代码定位策略"找到最相似功能的现有测试用例，读完整实现
- [ ] 确认测试数据准备方式（SQL fixture / API 调用 / mock）
- [ ] 确认断言模式（API 响应校验、数据库状态校验、页面元素校验）
- [ ] 确认 E2E 测试的登录态和 cookie 处理方式
- [ ] 复用现有的 test helper / utility

## 常见遗漏导致的问题

| 跳过的调研 | 导致的问题 |
|-----------|-----------|
| 没读现有 Service 实现 | 接口设计与现有模式不一致，review 时大改 |
| 没查表结构 DDL | 假设字段存在但实际没有，或字段名写错 |
| 没看前端页面代码 | 不知道现有交互模式，设计的交互与项目风格不统一 |
| 没确认打印/导出等特殊能力的实现方式 | 方案基于错误假设，如假设用 BrowserPrint 但实际用的是 URL 打开 |
| 没查已有的工具类 | 重复造轮子，review 时被要求改用现有实现 |

## 补充资料规范

当需求输入存在内容缺失（图片、表格、原型等），用户补充后必须固化到 proposal 中。

```markdown
## 补充资料

> ⚠️ 以下内容由用户口述/截图补充，非原始需求文档直接提取。
> 如与需求文档原文存在冲突，以需求文档为准，请用户确认差异点。

### {标题}（来源：需求文档 "{对应的原文标题}"）
{用户补充的内容，用文字描述图片/表格的关键信息}
```

规则：
1. **标注来源**：每条补充资料标注对应的原始标题，方便回溯
2. **冲突提醒**：章节顶部固定放置 ⚠️ 提醒，告知后续读者（包括 apply 阶段的 AI）注意版本一致性
3. **文字化**：图片内容必须转化为文字描述，不能只说"见截图"
4. **不修改原文引用**：补充资料是增量，不是替换
5. **冲突处理**：补充内容与已写方案矛盾时，用 `⚠️ 冲突` 标记，并在 design/specs 中同步修正
