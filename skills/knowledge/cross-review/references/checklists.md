# 审核检查清单（通用版）

> 供 self_review 入口使用。调用方可直接引用，也可在此基础上增删项目特定检查项。

---

## 一、方案/提案检查清单

### 1. 需求覆盖性
- [ ] 需求文档中的每个功能点是否都在方案中体现？
- [ ] 是否有过度设计（需求没要求但方案中出现）？
- [ ] 需求的优先级和范围是否正确理解？

### 2. 数据模型设计
- [ ] 表结构、字段类型、默认值是否合理？
- [ ] 业务唯一键是否明确且正确？
- [ ] 索引设计是否支持主要查询场景？
- [ ] 是否考虑了数据量增长后的性能？

### 3. 接口设计
- [ ] 接口路径和方法是否规范？
- [ ] 请求/响应参数是否完整？
- [ ] 分页、排序、过滤条件是否合理？

### 4. 测试场景完整性
- [ ] 功能场景：是否覆盖所有需求点？
- [ ] 边界场景：空值、空集合、边界值、超长字符串？
- [ ] 异常场景：DB 异常、外部服务异常、网络超时？
- [ ] 性能场景：大批量数据、分页边界？
- [ ] 每个场景的前置条件和预期结果是否具体可验证？

### 5. 兼容性与回退
- [ ] 是否影响现有功能？
- [ ] 大范围改造是否有功能开关或回退机制？
- [ ] 默认行为是否保持不变？

### 6. 国际化与时区
- [ ] 用户可见文本是否使用 i18n key，而非硬编码中文？
- [ ] 涉及时间展示/提交的字段是否经过时区转换处理？

## 二、代码检查清单

### 1. 功能完整性
- [ ] 是否实现了所有需求点/测试场景？
- [ ] 每个场景的预期结果是否满足？

### 2. 数据模型一致性
- [ ] 查询条件是否与索引设计匹配？
- [ ] 唯一性校验 key 是否与业务唯一键一致？
- [ ] 聚合/分组 key 是否正确？
- [ ] 逻辑删除是否正确处理？

### 3. 边界与异常处理
- [ ] 参数校验：null、空字符串、空集合？
- [ ] 数据不存在的场景是否处理？
- [ ] 外部调用异常是否有降级/重试？
- [ ] 事务边界是否正确？是否有长事务风险？

### 4. 性能
- [ ] 是否有 N+1 查询？
- [ ] 循环内是否有数据库操作？
- [ ] 批量操作是否分批处理？

### 5. SQL 与数据访问安全
- [ ] 是否存在 SQL 拼接风险？
- [ ] 动态查询条件是否处理了空值/空集合？
- [ ] 查询条件是否命中索引？
- [ ] 列表查询是否有分页或 LIMIT？
- [ ] UPDATE/DELETE 条件是否精确？

### 6. 安全性
- [ ] 用户输入是否经过校验和清洗？
- [ ] 接口是否有权限校验？
- [ ] 查询/修改是否校验数据归属（防越权）？
- [ ] 日志/响应中是否泄露敏感数据？
- [ ] 是否有硬编码密钥/密码？

### 7. 并发与幂等
- [ ] 先查后改模式是否有并发保护？
- [ ] 写操作接口是否支持幂等？
- [ ] 外部调用是否配置了合理超时？

### 8. 变更隔离
- [ ] 是否影响现有调用方？
- [ ] 默认行为是否保持不变？
- [ ] 大改造是否有回退机制？

### 9. 租户隔离与数据权限
- [ ] 查询条件是否包含 tenant_id（多租户隔离）？
- [ ] 是否评估了数据按用户隔离的需求（如只看自己创建的记录）？

### 10. 国际化与时区
- [ ] 用户可见文本是否使用 i18n key，而非硬编码中文？
- [ ] 涉及时间展示/提交的字段是否经过时区转换处理？
- [ ] 新增枚举/状态值是否同步维护多语言翻译？

### 11. 代码质量
- [ ] 方法长度 >30 行是否需要拆分？
- [ ] 嵌套 >3 层是否用卫语句？
- [ ] 是否有魔法值？
- [ ] 是否有可复用的现有工具类/枚举？

## 三、测试检查清单

### 1. 场景覆盖
- [ ] 测试用例是否完整覆盖所有需求/场景？
- [ ] 列出遗漏的场景

### 2. 测试数据
- [ ] 数据是否符合业务规则和数据库约束？
- [ ] 是否有数据依赖（需先准备关联数据）？
- [ ] 数据清理是否完整（不留脏数据）？

### 3. 预期结果充分性
- [ ] 断言是否覆盖关键响应字段？
- [ ] 需要验证数据库状态的用例是否有 DB 断言？

### 4. 边界与异常
- [ ] 是否覆盖空值、极值、异常输入？
- [ ] 是否覆盖外部依赖故障场景？

## 四、通用检查清单

> 适用于 skill、文档、设计方案、配置文件等非代码/非提案/非测试的内容审核。

### 1. 目标与定位
- [ ] 目标是否清晰？解决什么问题、服务谁？
- [ ] 与现有同类内容是否有职责重叠或冲突？
- [ ] 定位描述与实际能力是否匹配（不过度承诺、不遗漏关键能力）？

### 2. 结构完整性
- [ ] 必要的组成部分是否齐全（无遗漏章节/文件/配置）？
- [ ] 各部分之间的引用关系是否正确（路径、名称、链接）？
- [ ] 是否有声明了但未实现的功能或占位符？

### 3. 一致性
- [ ] 内部术语、命名、格式是否前后一致？
- [ ] 与项目已有规范/约定是否一致（命名风格、目录结构、文件格式）？
- [ ] 对外声明的接口/协议与实际行为是否一致？

### 4. 可维护性
- [ ] 修改某一处时，需要同步修改的地方是否可控（避免散弹式修改）？
- [ ] 配置和硬编码是否分离？环境特定内容是否可配置？
- [ ] 是否有清晰的扩展点（新增类型/场景时改动量是否合理）？

### 5. 可用性
- [ ] 首次使用者能否根据文档独立完成操作？
- [ ] 错误场景是否有明确的提示和恢复路径？
- [ ] 边界情况（空输入、异常输入、缺少依赖）是否处理？

### 6. 集成性
- [ ] 与上下游的调用关系是否明确且已建立？
- [ ] 是否存在"声称被调用但实际无调用方"的悬空设计？
- [ ] 依赖的外部工具/服务不可用时是否有降级方案？

### 7. 指令有效性（rules、AGENTS.md、prompt 模板）
- [ ] 需要 AI 主动执行的指令（如"读取某文件"、"检查某条件"），是否告诉了 AI 怎么验证自己已经做过了？反例：`必须读取 X 文件`；正例：`如果你的上下文中没有 X 的内容，先读取该文件`
- [ ] 文档中是否出现了 `fs_read`、`execute_bash`、`Read`、`Bash`、`Grep` 等具体工具函数名？如果有，改为描述意图（如"执行一条 shell 命令"、"读取以下文件"）
