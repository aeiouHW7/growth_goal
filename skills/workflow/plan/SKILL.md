# plan - 需求规划和任务分解

在 propose 之前使用，帮助将模糊的需求转化为清晰的实现方案。

## 触发条件

- 用户说"规划 XX 功能"、"讨论需求"
- 需求复杂，不确定如何拆分
- 需要评估工作量和优先级

## 前置检查

无前置要求（可随时使用）

---

## 执行流程

### 1. 需求定级

判断需求复杂度，决定规划深度：

| 级别 | 判断条件 | 规划产出 | 典型例子 |
|------|---------|---------|---------|
| **简单** | 单文件、无状态变更 | 简要描述 + 验收标准 | 加字段、改 UI、修 bug |
| **中等** | 涉及状态变更或新流程 | 功能描述 + 状态图 + 场景列表 | 加审批流程、新操作 |
| **复杂** | 新实体、跨模块联动 | 完整规划文档（见 Phase 2） | 新功能模块、架构变更 |

**输出示例**：
```
🎯 需求定级：中等
- 涉及订单状态变更（待支付 → 已支付）
- 影响模块：订单中心、支付中心
- 预计工作量：2-3 天

建议规划深度：功能描述 + 状态图 + 核心场景
```

### 2. 需求探索（苏格拉底式对话）

通过结构化提问帮助澄清需求：

**核心问题**（每次最多 3 个）：

**业务背景**：
- 为什么要做？当前痛点是什么？
- 能量化吗？（如：减少 50% 操作时间）

**用户场景**：
- 谁在什么场景下使用？
- 当前怎么做的？新方案有什么不同？

**核心实体**：
- 涉及哪些业务对象？（用户、订单、商品...）
- 它们的状态怎么流转？

**边界**：
- 做什么、不做什么？
- 和现有功能怎么衔接？

**异常场景**：
- 并发怎么办？失败怎么回滚？
- 数据不一致怎么处理？

**示例对话**：
```
用户: "添加用户积分功能"

AI: 
1. 积分何时增加/扣减？订单完成时？还是支付时？
2. 积分有有效期吗？过期如何处理？
3. 用户查看积分的入口在哪？个人中心？订单详情？

[绘制初步流程图]
```

### 3. 信息收集闭环

根据级别判断是否收集完整：

**简单需求**：
- ✅ 明确的验收标准
- ✅ 影响范围清晰

**中等需求**：
- ✅ 功能描述清晰
- ✅ 状态流转明确
- ✅ 核心场景覆盖

**复杂需求**：
- ✅ 业务目标量化（P.A.M：问题-方案-度量）
- ✅ 实体和状态机完整
- ✅ 交互流程清晰
- ✅ 异常场景覆盖

### 4. 输出规划文档

根据复杂度生成规划：

**简单需求**（直接进入 propose）：
```markdown
## 需求：添加用户头像上传

**验收标准**：
- 支持上传 jpg/png，最大 2MB
- 头像显示在个人中心
- 上传失败有错误提示

**影响文件**：
- frontend/src/components/UserProfile.tsx
- backend/src/routes/upload.ts
```

**中等需求**（沉淀到 90_PLANNING/）：
```markdown
# 用户积分功能规划

## 目标
减少用户流失率，提升复购（当前复购率 30%，目标 45%）

## 核心流程
[状态图]
积分累积：订单完成 → 增加积分
积分使用：下单时抵扣 → 扣减积分
积分过期：180天后 → 清零

## 核心场景
1. 用户下单获得积分
2. 用户使用积分抵扣
3. 积分过期处理

## 异常处理
- 并发扣减：乐观锁
- 订单退款：积分回滚
```

**复杂需求**（完整 PRD）：
```markdown
# XX 功能完整规划

## O - 目标
**P (Problem)**: 当前问题描述
**A (Approach)**: 解决方案
**M (Metrics)**: 度量指标

## A - 架构
**实体模型**：[实体关系图]
**状态机**：[状态流转图]

## I - 交互
**端点**：Web/API/Mobile
**操作流程**：[交互流程图]

## S - 场景
**正常场景**：...
**异常场景**：...
**边界情况**：...
```

### 5. 任务拆分建议

根据规划，建议拆分为多个 propose：

**拆分原则**：
- 每个 propose 聚焦单一能力
- 按依赖关系排序
- 控制每个 propose 的复杂度

**示例**：
```
📋 建议拆分为 3 个 propose：

Phase 1: 基础积分系统
- propose: add-points-core
- 内容：积分表、增减 API、查询接口
- 复杂度：中等
- 预计：2 天

Phase 2: 积分使用
- propose: add-points-redemption
- 内容：下单抵扣、积分回滚
- 依赖：Phase 1
- 复杂度：中等
- 预计：1.5 天

Phase 3: 积分过期
- propose: add-points-expiry
- 内容：定时任务、过期通知
- 依赖：Phase 1
- 复杂度：简单
- 预计：1 天
```

### 6. 沉淀规划文档

**简单需求**：不沉淀（直接 propose）

**中等/复杂需求**：沉淀到 `90_PLANNING/`

```bash
mkdir -p 90_PLANNING/v1.0-user-points
cat > 90_PLANNING/v1.0-user-points/planning.md <<EOF
# 用户积分功能规划
[规划内容]
EOF
```

### 7. 建议下一步

```
✅ 规划完成

📋 建议拆分为 3 个 propose（见上）

💡 下一步:
1. 如果规划需要调整，继续讨论
2. 如果规划 OK，创建第一个 propose：
   "创建提案 add-points-core"
```

## 输出示例

### 简单需求规划

```
## Planning: 添加用户头像上传

🎯 级别：简单
📁 影响：1 个前端组件 + 1 个后端路由

验收标准：
- 支持 jpg/png，最大 2MB
- 显示在个人中心
- 上传失败有提示

💡 建议：直接创建 propose
   "创建提案 add-user-avatar"
```

### 复杂需求规划

```
## Planning: 用户积分系统

🎯 级别：复杂
📁 影响：订单中心、支付中心、用户中心

核心流程：
  订单完成 → 增加积分
  下单抵扣 → 扣减积分
  180天后 → 积分过期

📋 拆分建议：3 个 propose
  Phase 1: add-points-core（2天）
  Phase 2: add-points-redemption（1.5天）
  Phase 3: add-points-expiry（1天）

📝 已沉淀：90_PLANNING/v1.0-user-points/

💡 下一步：
   "创建提案 add-points-core"
```

---

## 护栏

- 每次最多问 3 个问题（避免信息过载）
- 必须给出具体选项（禁止宽泛问题）
- 使用 ASCII 图表辅助梳理
- 复杂需求必须沉淀文档
- 不在此阶段编写代码

---

## 与 propose 的关系

```
plan（规划阶段）
  ↓
分解为多个小需求
  ↓
propose（逐个创建提案）
  ↓
apply（实现）
```

**何时跳过 plan**：
- 需求已经很清晰
- 单一功能，无需拆分
- 紧急 bug 修复

**何时必须 plan**：
- 需求模糊，边界不清
- 涉及多个模块
- 需要评估工作量
