# 辩证思考模式库

本文档提供结构化的思考模式和模板，帮助在 Explore 和 Compare 阶段更系统地分析问题。

---

## 1. SWOT 分析模式

**适用场景**: 评估单个方案的全貌，识别内部优劣和外部机会威胁

**模板**:
```
方案: [方案名称]

┌─────────────────────────────────────────────┐
│            SWOT 分析                        │
├──────────────────┬──────────────────────────┤
│ Strengths (优势) │ Weaknesses (劣势)        │
│ - [内部优势 1]   │ - [内部劣势 1]           │
│ - [内部优势 2]   │ - [内部劣势 2]           │
│ - [内部优势 3]   │ - [内部劣势 3]           │
├──────────────────┼──────────────────────────┤
│ Opportunities    │ Threats (威胁)           │
│ (机会)           │                          │
│ - [外部机会 1]   │ - [外部威胁 1]           │
│ - [外部机会 2]   │ - [外部威胁 2]           │
│ - [外部机会 3]   │ - [外部威胁 3]           │
└──────────────────┴──────────────────────────┘
```

**使用步骤**:
1. **Strengths**: 方案的内在优点（技术优势、性能优势、已有基础）
2. **Weaknesses**: 方案的内在缺点（复杂度、技术债、学习曲线）
3. **Opportunities**: 外部有利因素（社区趋势、团队技能、未来扩展）
4. **Threats**: 外部不利因素（技术过时风险、竞品压力、依赖风险）

**示例**:
```
方案: 使用 GraphQL 替代 REST

Strengths:
- 客户端可按需查询，减少过度获取
- 强类型 schema，自动生成文档
- 单端点，减少 API 版本管理

Weaknesses:
- 学习曲线陡峭，团队需要培训
- 查询复杂度不可控，可能导致 N+1 问题
- 缓存策略比 REST 复杂

Opportunities:
- 前端团队已有 GraphQL 经验
- 移动端需要灵活查询，GraphQL 优势明显
- 行业趋势向 GraphQL 转移

Threats:
- 后端团队不熟悉，初期生产力下降
- 工具链不如 REST 成熟
- 性能优化需要更多经验
```

## 2. 五个为什么 (5 Whys)

**适用场景**: Question 阶段深入挖掘问题根因，避免解决表面问题

**模板**:
```
问题: [表面问题]

为什么 1: [问题的直接原因]
  ↓
为什么 2: [原因的原因]
  ↓
为什么 3: [更深层的原因]
  ↓
为什么 4: [接近根因]
  ↓
为什么 5 (根因): [真正需要解决的问题]
```

**使用步骤**:
1. 陈述表面问题
2. 问"为什么会发生？"
3. 对答案继续问"为什么？"
4. 重复至少 5 次，直到找到根因
5. 针对根因设计解决方案

**示例**:
```
问题: API 响应慢

为什么 1: 因为数据库查询慢
  ↓
为什么 2: 因为查询没有索引
  ↓
为什么 3: 因为数据库设计时没有考虑查询模式
  ↓
为什么 4: 因为设计时缺少性能需求的明确定义
  ↓
为什么 5 (根因): 因为需求分析阶段没有涉及性能指标

解决方案: 不仅加索引，更要建立性能需求分析流程
```

## 3. 决策矩阵 (Decision Matrix)

**适用场景**: Compare 阶段量化对比多个方案，减少主观判断

**模板**:
```
| 维度          | 权重 | 方案 A | 方案 B | 方案 C |
|--------------|------|--------|--------|--------|
| [维度 1]     | 30%  | 8(2.4) | 6(1.8) | 9(2.7) |
| [维度 2]     | 25%  | 7(1.75)| 9(2.25)| 5(1.25)|
| [维度 3]     | 20%  | 6(1.2) | 8(1.6) | 7(1.4) |
| [维度 4]     | 15%  | 9(1.35)| 5(0.75)| 8(1.2) |
| [维度 5]     | 10%  | 7(0.7) | 7(0.7) | 6(0.6) |
|--------------|------|--------|--------|--------|
| **加权总分** |      | **7.4**| **7.1**| **7.15**|
```

**使用步骤**:
1. 列出所有关键评估维度
2. 为每个维度分配权重（总和 100%）
3. 为每个方案在每个维度打分（1-10 分）
4. 计算加权分数：分数 × 权重
5. 汇总得出总分

**权重设定指引**:
- **关键约束**: 30-40%（如预算限制、截止时间）
- **重要目标**: 20-30%（如性能、可维护性）
- **次要因素**: 10-20%（如美观度、文档）
- **可选因素**: 5-10%（如新颖性、趋势）

**示例**:
```
技术栈选择: React vs Vue vs Svelte

| 维度          | 权重 | React  | Vue    | Svelte |
|--------------|------|--------|--------|--------|
| 团队熟悉度    | 35%  | 9(3.15)| 6(2.1) | 3(1.05)|
| 生态完整度    | 25%  | 9(2.25)| 8(2.0) | 5(1.25)|
| 性能         | 20%  | 7(1.4) | 8(1.6) | 9(1.8) |
| 学习曲线     | 15%  | 5(0.75)| 8(1.2) | 7(1.05)|
| 打包体积     | 5%   | 6(0.3) | 7(0.35)| 9(0.45)|
|--------------|------|--------|--------|--------|
| **总分**     |      | **7.85**| **7.25**| **5.6** |

结论: React 胜出，主要因为团队熟悉度和生态权重高
```

## 4. 正反辩论 (Pros/Cons Deep Dive)

**适用场景**: 深度分析单个方案，全面列举优缺点

**模板**:
```
方案: [方案名称]

┌─────────────────────────────────────────────┐
│  正方观点 (Pros)       vs  反方观点 (Cons)  │
└─────────────────────────────────────────────┘

正方 Round 1: [最强优点]
  └─ 反方反驳: [这个优点的局限性]
      └─ 正方辩护: [如何缓解这个局限]

正方 Round 2: [第二优点]
  └─ 反方反驳: [局限性]
      └─ 正方辩护: [缓解]

反方 Round 1: [最强缺点]
  └─ 正方反驳: [这个缺点可接受的理由]
      └─ 反方质疑: [为什么不可接受]

反方 Round 2: [第二缺点]
  └─ 正方反驳: [可接受理由]
      └─ 反方质疑: [不可接受理由]
```

**使用步骤**:
1. 列出方案的最强优点
2. 为每个优点找出反驳（优点的局限性）
3. 为反驳提供辩护（如何缓解）
4. 列出方案的最强缺点
5. 为每个缺点找出正方解释（为何可接受）
6. 继续质疑正方解释

**示例**:
```
方案: 采用微服务架构

正方 Round 1: 独立部署，每个服务可单独发布
  └─ 反方反驳: 增加运维复杂度，需要容器编排
      └─ 正方辩护: 可以用 K8s 自动化编排，长期降低成本

正方 Round 2: 技术栈多样性，每个服务可选最优技术
  └─ 反方反驳: 增加团队学习负担，招聘困难
      └─ 正方辩护: 可限制为 2-3 种主流技术栈

反方 Round 1: 分布式事务复杂，数据一致性难保证
  └─ 正方反驳: 可以用 Saga 模式或事件溯源
      └─ 反方质疑: 这些模式本身也很复杂，团队是否有能力？

反方 Round 2: 调试困难，跨服务追踪日志复杂
  └─ 正方反驳: 可以用分布式追踪系统（Jaeger）
      └─ 反方质疑: 又增加一个新系统，运维负担更重
```

## 5. 阶段路线图 (Phased Roadmap)

**适用场景**: Decide 阶段规划渐进式实施，降低一次性切换风险

**模板**:
```
最终目标方案: [理想方案]

┌─────────────────────────────────────────────┐
│              阶段路线图                      │
└─────────────────────────────────────────────┘

Phase 0: 现状
  - [当前状态描述]
  - 痛点: [主要问题]

Phase 1: [时间] - [阶段名称]
  - 目标: [这个阶段解决什么]
  - 实施: [具体做什么]
  - 风险: [主要风险]
  - 退出标准: [如何判断可进入下一阶段]

Phase 2: [时间] - [阶段名称]
  - 目标: ...
  - 实施: ...
  - 风险: ...
  - 退出标准: ...

Phase 3: [时间] - [最终状态]
  - 目标: ...
  - 实施: ...
  - 预期收益: [量化收益]
```

**使用步骤**:
1. 定义最终目标状态
2. 定义当前状态（Phase 0）
3. 将中间过程拆分为 2-4 个阶段
4. 每个阶段设置明确的退出标准
5. 确保每个阶段都能独立产生价值

**示例**:
```
最终目标: 完全微服务化的架构

Phase 0: 单体应用
  - 所有功能在一个 Rails 应用中
  - 痛点: 部署慢、扩展困难

Phase 1: Q1 - 读写分离
  - 目标: 数据库性能提升
  - 实施: 主从复制、读操作走从库
  - 风险: 主从延迟
  - 退出标准: 读写 QPS 提升 50%

Phase 2: Q2 - 关键服务拆分
  - 目标: 拆分出订单服务（最核心、变化最频繁）
  - 实施: 订单模块独立为服务，API Gateway 路由
  - 风险: 分布式事务
  - 退出标准: 订单服务独立部署成功

Phase 3: Q3-Q4 - 全面微服务化
  - 目标: 所有核心模块服务化
  - 实施: 拆分用户、支付、库存服务
  - 预期收益: 部署时间从 1h 降至 5min
```

## 模式选择指南

| 阶段    | 推荐模式                          | 目的                     |
|---------|-----------------------------------|--------------------------|
| Question| 五个为什么                         | 找到问题根因             |
| Explore | SWOT（针对每个方案）              | 全面评估单个方案         |
| Compare | 决策矩阵 + 正反辩论                | 量化对比 + 深度分析      |
| Decide  | 阶段路线图                         | 规划渐进式实施           |

**综合使用建议**:
- 不必每次都用所有模式，根据问题复杂度选择
- 简单决策：用决策矩阵足够
- 复杂决策：组合使用多个模式
- 团队决策：SWOT + 正反辩论可促进充分讨论
