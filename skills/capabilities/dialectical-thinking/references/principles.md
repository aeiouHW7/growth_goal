# 辩证思考核心原则

本文档包含辩证思考的核心原则，帮助在 Question 阶段更深入地质疑和分析问题。

---

## 1. 第一性原理 (First Principles)

**定义**: 回到事物最基本的真理，从基础开始推导，而不是基于类比或惯例。

**如何使用**:
1. 问"为什么"直到无法再问（追溯到最基本的假设）
2. 识别哪些是真理，哪些是约定俗成
3. 从真理出发重新构建解决方案

**示例**:
- ❌ 类比思维: "其他公司都用 Kubernetes，所以我们也用"
- ✅ 第一性原理: "我们需要容器编排吗？→ 需要。为什么？→ 因为有多个服务需要部署。能否用更简单的方式？→ Docker Compose 足够"

**适用场景**: 挑战行业惯例、避免过度工程、技术选型

---

## 2. 逆向思维 (Inversion)

**定义**: 从相反方向思考问题——不是"如何成功"，而是"如何失败"，然后避免失败因素。

**如何使用**:
1. 列出"如何让这个方案彻底失败"的所有可能
2. 将每个失败因素转化为需要避免的风险
3. 设计方案时主动规避这些风险

**示例**:
- 正向: "如何设计高可用系统？"
- 逆向: "什么会导致系统不可用？→ 单点故障、依赖挂掉、网络分区 → 方案必须处理这些"

**适用场景**: 风险识别、架构设计、预防性设计

---

## 3. 魔鬼代言人 (Devil's Advocate)

**定义**: 故意为相反观点辩护，挑战自己的偏好方案，寻找其弱点。

**如何使用**:
1. 选定你倾向的方案
2. 假设你必须反对它，找出所有问题
3. 为其他方案辩护，找出它们的优点
4. 客观评估所有观点

**示例**:
- 倾向方案: "用 React"
- 魔鬼代言: "为什么不用 Vue？→ Vue 更简单、学习曲线低、打包更小。React 的优势真的值得复杂度吗？"

**适用场景**: 避免确认偏误、技术选型、团队决策

---

## 4. 质疑假设 (Challenge Assumptions)

**定义**: 识别并挑战所有隐含假设，尤其是那些"理所当然"的前提。

**如何使用**:
1. 列出所有假设（包括显式和隐式的）
2. 逐个质疑："这个假设必须成立吗？"
3. 测试假设不成立时会发生什么
4. 重新定义不依赖不合理假设的问题

**示例**:
- 假设: "用户需要实时更新"
- 质疑: "真的需要实时（<100ms）还是准实时（<5s）就够？→ 测试发现 5s 延迟用户无感知 → 可以用轮询而非 WebSocket"

**适用场景**: 需求分析、过度设计预防、简化方案

---

## 5. 奥卡姆剃刀 (Occam's Razor)

**定义**: 如无必要，勿增实体。在多个方案中，选择假设最少、最简单的那个。

**如何使用**:
1. 列出所有方案
2. 识别每个方案引入的额外复杂度
3. 问"这个复杂度解决了什么真实问题？"
4. 优先选择最简单且满足需求的方案

**示例**:
- 复杂方案: "微服务 + Kafka + Redis + K8s"
- 奥卡姆剃刀: "单体应用 + PostgreSQL 能满足当前 1000 用户的需求吗？→ 能 → 先用简单方案"

**适用场景**: 避免过度设计、MVP 设计、技术栈选择

---

## 6. 二阶思维 (Second-Order Thinking)

**定义**: 不仅考虑决策的直接后果（一阶），还考虑后续连锁反应（二阶、三阶）。

**如何使用**:
1. 一阶: 这个决策的直接结果是什么？
2. 二阶: 然后会发生什么？
3. 三阶: 再然后呢？
4. 评估整个因果链

**示例**:
- 一阶: "选择 NoSQL → 写入更快"
- 二阶: "没有 JOIN → 需要在应用层聚合数据 → 代码复杂度增加"
- 三阶: "代码复杂 → 维护成本高 → 团队生产力下降"

**适用场景**: 架构决策、技术债评估、长期规划

---

## 7. 基线对比 (Baseline Comparison)

**定义**: 不要孤立评估方案，而是与现状或最简单方案对比，明确增量价值。

**如何使用**:
1. 定义基线（通常是"什么都不做"或"最简单方案"）
2. 每个新方案都与基线对比
3. 明确回答："相比基线，这个方案的增量价值是什么？增量成本是什么？"

**示例**:
- 基线: "不做缓存"
- 方案 A (Redis): "增量价值: 性能提升 10x；增量成本: 部署运维复杂度"
- 方案 B (In-memory): "增量价值: 性能提升 8x；增量成本: 重启丢失缓存"
- 评估: B 的性价比更高

**适用场景**: ROI 评估、渐进式改进、避免盲目优化

---

## 使用建议

- **Question 阶段**: 重点使用原则 1, 4, 7（第一性原理、质疑假设、基线对比）
- **Explore 阶段**: 重点使用原则 2, 5（逆向思维、奥卡姆剃刀）
- **Compare 阶段**: 重点使用原则 3, 6（魔鬼代言人、二阶思维）
- **Decide 阶段**: 综合运用所有原则

**注意**: 这些原则不是公式，而是思维工具。根据实际情况灵活选用，不必每次都用全部。
