# 辩证思考实战案例

本文档提供真实的辩证思考案例，展示完整的四阶段工作流在实际决策中的应用。

---

## 案例 1: ACE Engine 目录结构设计

**背景**: AI-Coding-Engine 项目启动时，需要决定根目录的组织方式。参考项目 ai-drive-engine 使用了数字前缀（10_DOCS, 20_FE 等），但是否应该照搬？

### Stage 1: Question - 质疑问题

**需求本身是否合理？**

表面需求："设计 ACE Engine 的目录结构"

深入质疑：真正的问题是什么？
- ✅ 不仅是"怎么组织目录"
- ✅ 而是"如何让 AI 和人类都能快速理解项目结构"
- ✅ 以及"如何平衡组织性和语义化"

**有哪些隐含假设？**

1. **假设 1**: 数字前缀有助于排序
   - 质疑: 现代编辑器都支持自定义排序，数字前缀是否必要？
   - 验证: 确实，在文件树中数字前缀强制优先级（文档 > 代码 > 规划）

2. **假设 2**: ai-drive-engine 的方式是最优的
   - 质疑: ai-drive 是特定业务项目，ACE 是通用框架，需求不同
   - 重新思考: ACE 的核心是根目录（rules, skills, templates），不是 Domain 内部

3. **假设 3**: 所有用户都喜欢数字前缀
   - 质疑: 有些人觉得不语义化、不直观
   - 平衡: 可以在关键位置保留，其他位置语义化

**问题重定义**:

❌ 原问题: "是否使用数字前缀目录？"  
✅ 新问题: "在 ACE Engine 根目录设计中，如何平衡强制优先级（数字）和语义清晰（命名），使得 AI 和人类都能高效导航？"

---

### Stage 2: Explore - 探索方案

#### 方案 A: 完全数字化（ai-drive 模式）

**核心思路**: 所有目录都加数字前缀，强制优先级

**结构示例**:
```
10_RULES/
20_SKILLS/
30_TEMPLATES/
40_DOMAINS/
90_DOCS/
```

**关键技术**: 数字编码优先级规则
- 10-30: 核心资产（规则、技能、模板）
- 40-80: 业务内容
- 90+: 元数据、文档

---

#### 方案 B: 完全语义化（去掉数字）

**核心思路**: 纯语义化命名，依赖约定和文档

**结构示例**:
```
rules/
skills/
templates/
domains/
docs/
```

**关键技术**: 
- README 驱动：在根目录 README 明确优先级
- 约定大于配置：团队约定哪些是核心目录

---

#### 方案 C: 混合模式（语义为主，关键保留数字）

**核心思路**: 语义化为主体，仅在强调优先级的地方用数字

**结构示例**:
```
rules/           # 纯语义化
  ├── system/
  └── coding/
skills/          # 纯语义化
  ├── system/
  └── workflow/
templates/       # 纯语义化
domains/         # 纯语义化
  └── example-domain/
      ├── docs/         # 语义化
      ├── frontend/     # 语义化
      ├── backend/
      └── planning/     # 语义化
```

**关键技术**:
- 根目录 100% 语义化（清晰、专业）
- Domain 内部全部语义化（docs, planning, frontend, backend）

---

### Stage 3: Compare - 对比分析

| 维度          | 方案 A (全数字) | 方案 B (全语义) | 方案 C (混合) |
|--------------|-----------------|-----------------|---------------|
| **AI 导航性** | ★★★ 强制优先   | ★★☆ 需要学习   | ★★★ 平衡      |
| **人类可读性**| ★☆☆ 不直观      | ★★★ 最直观     | ★★★ 直观      |
| **专业度**    | ★★☆ 像脚手架   | ★★★ 正式项目   | ★★★ 正式      |
| **一致性**    | ★★★ 高度一致   | ★★★ 语义一致   | ★★☆ 需要约定  |
| **扩展性**    | ★★☆ 数字冲突   | ★★★ 无限制     | ★★★ 灵活      |
| **维护成本**  | ★★★ 规则清晰   | ★☆☆ 需要文档   | ★★☆ 中等      |

#### 关键权衡点

**权衡 1: AI 导航 vs 人类可读**
- 方案 A: AI 友好（数字优先级明确），但人类觉得不专业
- 方案 B: 人类友好，但 AI 需要学习优先级
- 方案 C: 两者平衡，根目录语义（面向人），Domain 内部数字（面向 AI）

**权衡 2: 一致性 vs 灵活性**
- 方案 A: 高度一致，但所有目录都被数字"污染"
- 方案 B: 完全灵活，但缺少约束，可能混乱
- 方案 C: 分层一致性，根目录纯净，Domain 内部有序

**权衡 3: 通用框架 vs 业务项目**
- ACE Engine 是**框架**，根目录是给所有用户看的（应该专业、语义化）
- Domain 是**业务项目**，内部是给开发者用的（可以用数字强化优先级）

---

### Stage 4: Decide - 决策推荐

#### 推荐方案: 方案 C（混合模式）

**推荐理由**:

1. **根目录代表框架形象**  
   - ACE Engine 是通用框架，根目录是"门面"
   - 语义化命名（rules, skills, templates）更专业、更清晰
   - 数字前缀在根目录显得像"脚手架项目"而非成熟框架

2. **Domain 内部保留优先级提示**  
   - 10_DOCS: 强调文档优先（知识驱动的 ETHOS 核心）
   - 90_PLANNING: 强调规划阶段的重要性
   - frontend/backend: 开发目录语义化即可，无需强制排序

3. **兼顾 AI 和人类**  
   - AI: 在 Domain 内部有数字引导，快速定位文档和规划
   - 人类: 根目录一眼就懂，不需要"翻译"数字含义

4. **参考行业最佳实践**  
   - Next.js: `pages/`, `components/` (语义化)
   - Nx Monorepo: `libs/`, `apps/` (语义化)
   - 成熟框架很少在根目录用数字前缀

**注意事项**:

- **约定文档化**: 在 ARCHITECTURE.md 中明确说明为什么 Domain 内部用数字
- **团队培训**: 新成员可能疑惑"为什么有的地方有数字，有的没有"
- **迁移成本**: 如果未来决定全面去掉数字，需要批量重命名（但影响小）

**不推荐的原因**:

- **方案 A（全数字）**: 
  - ACE 是框架，根目录应该"高大上"，数字前缀降低专业度
  - 用户看到 `10_RULES/` 会问"为什么是 10？20 在哪？"

- **方案 B（全语义）**: 
  - 失去了 ai-drive 的优秀经验（数字强化文档优先）
  - Domain 内部没有优先级提示，AI 需要更多指引

**后续行动**:

1. 更新 `ARCHITECTURE.md`，记录这个决策和理由
2. 创建 Domain 模板时，默认使用混合模式
3. 在 `.claude/memory/decisions.md` 记录这个决策

## 案例 2: ace-flow 启动模式设计

**背景**: 需要设计 ace-flow（一键启动环境）的实现方式。参考 ai-drive 有手工 dev.sh 脚本，但 ACE 应该更智能。

### Stage 1: Question

**需求重定义**:  
❌ "如何启动前后端？"  
✅ "如何在开发和生产环境下，自动识别依赖并用最优方式启动？"

**关键假设**:
- 假设: "必须用 Docker Compose 统一管理"
  - 质疑: 开发时用 Docker 会失去热重载，影响效率
  - 结论: 需要区分开发和生产模式

### Stage 2: Explore

**方案 A**: 纯 Docker Compose（统一但不灵活）  
**方案 B**: ace-flow 智能编排（读取 domain.yaml，自动决策）  
**方案 C**: 双模式脚本（dev.sh + prod.sh）

### Stage 3: Compare

| 维度 | 方案 A | 方案 B | 方案 C |
|------|--------|--------|--------|
| 开发体验 | ★☆☆ 无热重载 | ★★★ 智能选择 | ★★★ 本地热重载 |
| 一致性 | ★★★ 高 | ★★☆ 中 | ★☆☆ 两套配置 |
| 实现复杂度 | ★★★ 简单 | ★☆☆ 复杂 | ★★☆ 中等 |

### Stage 4: Decide

**推荐**: 方案 C（双模式脚本）

**理由**:
- 开发效率优先：热重载是刚需
- 实现成本合理：不需要复杂的智能编排
- 清晰明确：dev 和 prod 职责分明

**实施**:
- `npm run ace:up` → 启动基础设施（MySQL/Redis Docker）
- `npm run dev:frontend` → 本地启动前端（热重载）
- `npm run dev:backend` → 本地启动后端（热重载）
- `npm run prod:up` → Docker Compose 启动全部

## 案例 3: Skill 分层管理设计

**背景**: ACE Engine 需要管理大量 Skills，如何组织它们？

### Stage 1: Question

**问题重定义**:  
✅ "如何让 AI 快速找到合适的 Skill，同时让用户能管理自定义 Skill？"

**关键洞察**:
- 不同 Skill 的作用域不同（系统级 vs 业务级）
- 不同 Skill 的复用性不同（通用 vs 特定项目）

### Stage 2: Explore

**方案 A**: 平铺（所有 Skills 在 skills/ 下）  
**方案 B**: 三层分类（system / workflow / domain-specific）  
**方案 C**: 两层分类（system / coding）

### Stage 3: Compare

权衡核心是**清晰度 vs 复杂度**

### Stage 4: Decide

**推荐**: 方案 B（三层分类）

**理由**:
- system: 基础设施 Skills（ace-doctor, ace-init 等）
- workflow: 通用工作流 Skills（dialectical-thinking, openspec 等）
- domain-specific: 每个 Domain 可以有自己的 Skills

**实施**:
```
skills/
├── system/           # 系统级（ace-*）
├── workflow/         # 工作流（dialectical-thinking, code-review 等）
└── coding/           # 编码类（目前 dialectical-thinking 归这里）
```

## 案例总结

这三个案例展示了辩证思考的核心价值：

1. **避免直接执行**: 不是用户说"设计目录"就立刻给结构，而是先质疑需求
2. **多方案探索**: 每次至少2-3种方案，避免路径依赖
3. **理性权衡**: 用表格、原则、模式系统化对比
4. **有理有据**: 推荐时明确说出"为什么"，而不是"我觉得"

**何时使用辩证思考**:
- ❌ 不适合: "帮我修复这个 bug" → 直接解决
- ✅ 适合: "我们应该用微服务吗？" → 辩证分析
- ✅ 适合: "如何设计 API？" → 辩证探索
- ✅ 适合: "选择 React 还是 Vue？" → 辩证对比

**学到的经验**:
- Question 阶段往往能重新定义问题，避免解决错误的问题
- Explore 阶段强迫考虑多种可能性，打破思维定势
- Compare 阶段量化分析让决策更客观
- Decide 阶段必须给出明确推荐，但诚实列出风险

**下次使用时**:
- 如果时间有限，可以快速过一遍四阶段（每阶段 2-3 分钟）
- 如果决策重大，深入使用参考资料（principles, patterns）
- 如果团队讨论，用 Compare 阶段的表格促进客观对话
