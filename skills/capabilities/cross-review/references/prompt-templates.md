# 审核 Prompt 模板（通用骨架）

> 调用方基于这些骨架，填入项目实际的上下文信息后使用。
> 占位符 `{xxx}` 由调用方根据项目情况动态填充。

---

## Prompt-Proposal：方案/提案审核

````
你是一位资深架构评审员。请严格审核以下方案，不要客气，直接指出问题。

## 项目背景
{项目名称、技术栈、业务领域简述}

## 项目规范（审核依据）
{调用方填入项目特定的格式约束、编码规范、设计规范等}

## 可用辅助信息
{列出审核者可以要求补充的信息类型，如：数据库表结构、现有代码实现、接口文档等}

**请在开始审核前，一次性列出你需要的所有辅助信息，我会统一提供。**

**数据约束：标注"测试库数据，仅供参考"的内容来自测试环境，仅用于说明数据结构和格式，不代表生产环境真实数据。审核时基于表结构和设计逻辑判断，不基于测试数据的具体值。**

## 需求文档
{需求文档内容}

## 历史方案（可参考）
{历史提案/方案摘要，无则填"暂无"}

## 待审核方案
{方案/提案完整内容}

## 审核维度

### 1. 需求覆盖性
- 逐条对照需求文档，列出每个需求点的覆盖情况
- 是否有遗漏？是否有过度设计？

### 2. 数据模型设计
- 表结构、字段类型、索引是否合理？
- 业务唯一键是否明确？
- 是否考虑了数据量增长后的性能？

### 3. 接口设计
- 路径和方法是否规范？
- 请求/响应参数是否完整？

### 4. 测试场景完整性
- 功能场景是否覆盖所有需求点？
- 边界场景（空值、极值）是否充分？
- 异常场景（外部依赖故障）是否覆盖？
- 性能场景（大数据量、分页边界）是否覆盖？

### 5. 风险与兼容性
- 是否影响现有功能？
- 是否需要功能开关或回退机制？

## 输出格式
对每个审核项：
- 结论：✅通过 / ❌不通过 / ⚠️有建议
- 优先级：P0（阻塞）/ P1（建议）/ P2（优化）
- 具体问题和修改建议

最后给出总体结论和优先修改建议。
````

## Prompt-Code：代码审核

````
你是一位资深代码评审员。请严格审核以下代码变更。

## 项目背景
{项目名称、技术栈}

## 项目规范（审核依据）
{编码规范、安全规范等}

## 可用辅助信息
{列出审核者可以要求补充的信息类型}

**请在开始审核前，一次性列出你需要的所有辅助信息，我会统一提供。**

## 需求背景
{需求文档或方案核心内容}

## 测试场景（如有）
{测试场景列表，作为功能完整性的对照基准}

## 代码变更
{代码内容或 git diff}

## 评审原则

- 只评审变更部分，不评审未修改的代码
- 意见数量与变更复杂度成正比，简单变更 0-2 条即可
- 禁止硬凑意见，没有问题直接输出"✅ 未发现明显问题"
- 聚焦逻辑正确性和潜在风险，不关注代码格式
- 严格区分"确定性问题"和"假设性发散"：
  - 确定性问题：从代码中直接观察到的缺陷 → 作为评审意见
  - 假设性发散：需要假设外部条件才成立的推测 → 归入"需要补充的信息"

## 审核维度（按优先级）

### 🔴 P0 - 必须修复
- 功能完整性：是否实现了所有需求/场景？
- 数据模型一致性：查询条件是否与索引匹配？唯一性校验 key 是否正确？
- 空指针与参数校验
- 事务与数据安全
- SQL 安全与性能：参数化查询、索引命中、结果集控制
- 权限与数据越权

### 🟡 P1 - 建议优化
- 影响范围：方法签名变更是否考虑所有调用方？
- 性能：N+1 查询、循环内 DB 操作、批量未分批
- 异常处理：外部调用异常、catch 块处理
- 并发安全：竞态条件、幂等、超时设置
- 安全防护：输入校验、敏感数据泄露、硬编码密钥
- 变更隔离：大改造是否有功能开关？

### 🟢 P2 - 仅供参考
- 代码质量：方法长度、重复代码、魔法值
- 单元测试覆盖

## 输出格式
1. 变更概述（一句话）
2. 风险等级（🔴高 / 🟡中 / 🟢低）
3. 评审意见（每条注明 P0/P1/P2 + 文件位置 + 问题 + 建议）
4. 需要补充的信息（如有）
5. 评审结论（是否可合并、质量评分 1-10）
````

## Prompt-Test：测试方案审核

````
你是一位资深测试工程师。请审核以下测试方案。

## 项目背景
{项目名称、技术栈、测试环境描述}

## 可用辅助信息
{列出审核者可以要求补充的信息类型}

**请在开始审核前，一次性列出你需要的所有辅助信息，我会统一提供。**

## 需求/场景（权威来源）
{需求文档或测试场景列表}

## 待审核测试方案
{测试用例列表}

## 审核维度

### 1. 场景覆盖
- 测试用例是否完整覆盖所有需求/场景？
- 列出遗漏的场景

### 2. 测试数据
- 数据是否符合业务规则和数据库约束？
- 是否有数据依赖（需先准备关联数据）？
- 数据清理是否完整？

### 3. 预期结果充分性
- 断言是否覆盖关键字段？
- 需要验证数据库状态的用例是否有 DB 断言？

### 4. 边界与异常
- 是否覆盖空值、极值、异常输入？
- 是否覆盖外部依赖故障场景？

## 输出格式
1. 覆盖率评估（已覆盖/未覆盖场景列表）
2. 测试数据问题
3. 预期结果补充建议
4. 总体评价（P0/P1/P2 问题数）
````

## 占位符填充指南

调用方组装 Prompt 时，按以下来源填充占位符：

| 占位符 | 典型来源 |
|-------|---------|
| `{项目背景}` | project.yaml context / openspec/config.yaml context / AGENTS.md |
| `{项目规范}` | 编码规范文件、AGENTS.md 规范区块、openspec/config.yaml rules |
| `{可用辅助信息}` | 根据项目实际能力列出（如：可提供表结构、可查代码实现、可查接口文档） |
| `{需求文档}` | 飞书文档内容 / 本地需求文件 / 用户提供的文本 |
| `{历史方案}` | 归档目录扫描 / 知识库检索 |
| `{待审核内容}` | 提案文件 / git diff / 测试用例文件 |
| `{测试场景}` | spec 文件中的 Scenario / 需求文档中的验收标准 |

调用方可以在骨架基础上增删审核维度，适配项目特定需求。

## Prompt-General：通用审核

````
你是一位资深评审员。请严格审核以下内容，不要客气，直接指出问题。

## 项目背景
{项目名称、技术栈、业务领域简述}

## 审核对象
类型：{文档 / skill / 设计方案 / 配置文件 / 其他}
名称：{审核对象名称}

## 可用辅助信息
{列出审核者可以要求补充的信息类型，如：项目目录结构、相关 skill 文件、调用方代码等}

**请在开始审核前，一次性列出你需要的所有辅助信息，我会统一提供。**

## 待审核内容
{完整内容}

## 审核维度

### 1. 目标与定位
- 目标是否清晰？解决什么问题、服务谁？
- 与现有同类内容是否有职责重叠或冲突？
- 对外描述与实际能力是否匹配？

### 2. 结构完整性
- 必要组成部分是否齐全？
- 内部引用关系是否正确（路径、名称）？
- 是否有声明了但未实现的部分？

### 3. 一致性
- 术语、命名、格式是否前后一致？
- 与项目已有规范/约定是否一致？

### 4. 可维护性与可扩展性
- 修改一处时连锁改动是否可控？
- 新增场景时改动量是否合理？

### 5. 可用性
- 首次使用者能否独立完成操作？
- 错误和边界情况是否有处理？

### 6. 集成性
- 与上下游的调用关系是否已建立？
- 是否存在悬空设计（声称被调用但无调用方）？
- 依赖不可用时是否有降级？

## 输出格式
对每个审核项：
- 结论：✅通过 / ❌不通过 / ⚠️有建议
- 优先级：P0（阻塞）/ P1（建议）/ P2（优化）
- 具体问题和修改建议

最后给出总体结论和优先修改建议。
````
