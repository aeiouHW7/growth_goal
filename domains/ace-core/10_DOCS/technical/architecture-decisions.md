# 架构决策日志 (ADR) - 环境管理策略

## 决策背景

在构建 ACE 引擎时，关于“环境初始化”应采用何种驱动模式，我们（产品 Owner 与 AI）进行了深度辩证。

## 方案对比

### 1. 纯脚本化 (Script-Heavy)
*   **利**：执行极其精确。
*   **弊**：黑盒化，产品经理难以维护，跨平台兼容性差。

### 2. 纯配置化 (Config-Only)
*   **利**：简洁透明，产品经理友好。
*   **弊**：执行不确定性高（AI 容易误读参数），处理复杂逻辑困难。

## 最终决策：声明式状态 + 自适应执行 (The ACE Way)

我们选择“取长补短”的中间路径：

### 1. 声明式配置 (`domain.yaml`)
用户只需声明“我需要什么”（如 `database: mysql`），而非“如何安装”（不写 `docker run`）。这保持了配置的极简。

### 2. 原子化 Skill (Executor)
将复杂的 Bash/Docker 脚本封装进标准化的 `skills/` 中。脚本是固定的、经过测试的，保证了执行的底座稳健。

### 3. 自适应巡检环 (Feedback Loop)
这是 ACE 的核心增强。执行流程不再是：`配置 -> 脚本`，而是：
1.  **感知 (Sense)**：检查当前系统环境（是否有 Docker、端口是否占用）。
2.  **决策 (Decide)**：基于环境现状，动态调整脚本参数。
3.  **执行 (Act)**：运行原子 Skill。
4.  **验证 (Verify)**：通过连接测试确认服务真正可用。

## 这种方案的优势

*   **简洁**：产品经理只需看一眼 YAML 就能理解项目底座。
*   **高效**：利用了 AI 的环境感知能力，解决了脚本“水土不服”的问题。
*   **安全**：通过闭环验证，避免了“假成功”导致后续开发阻塞。
