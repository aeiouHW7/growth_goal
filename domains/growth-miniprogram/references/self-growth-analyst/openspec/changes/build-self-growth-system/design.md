## Context

用户是31岁大厂产品经理，工作3年多。目标是成为中国前100的富人。有年龄焦虑，觉得自己"刚起步但已经偏大"。沟通风格要求：**只接受犀利直接，不安慰不废话**。

用户的核心需求：
1. 一个AI能透过他的碎碎念看到他自己看不到的东西
2. 持续监督他的过去、现在、将来
3. 逼他往"人上人"跃升

用户的反馈（关键）："整体都不太行，都不够深入，不是一个好的架构"

这说明之前的实现有根本性问题：
- 内容深度不够
- 架构不够扎实
- 产品经理语境不足

用户给了明确的目录结构规范，这是新的基础。

## Goals / Non-Goals

**Goals：**
- 严格按照用户给的目录结构构建系统
- 每个文件要有实际可执行的内容，不是泛泛的描述
- 深度要足够：一个新人看一遍就知道怎么填
- PM语境要具体，不能用通用理论

**Non-Goals：**
- 不是做通用产品，只服务这个具体用户
- 不是实时监控，只处理用户主动输入
- 不做无证据的评分

## Decisions

### Decision 1: 为什么用这个目录结构？

用户给的规范是经过思考的：
- `core/` 放宪法级别的内容（身份、检索、评分）
- `memories/` 只做数据存储，不含逻辑
- `rules/` 把所有"是否"的判断写死，避免模糊
- `skills_library/` 把问题模板化，减少每次生成的随意性
- `config/` 静态配置，不常改

这是对的。分离关注点，规则要硬不要软。

### Decision 2: 为什么评分"不完全自动化"？

用户的目标是"证据驱动"而不是"感觉驱动"。

如果评分是AI自动实时更新：
- AI可能产生"幻觉分数"
- 用户不会信
- 没有闭环

所以评分只在周/月/半年时显式计算，每次必须有2条以上证据，用户确认后才生效。

### Decision 3: 为什么问题要存在 skills_library/ 而不是每次生成？

如果每次临时生成问题：
- 质量不稳定
- 可能重复
- 没有积累

所以每个维度3-5个probe问题，固化到文件里。周考时随机抽。

### Decision 4: 为什么 memory_retrieval.md 要限制token？

LLM上下文有限。

如果每次加载所有历史，上下文会爆。

所以限制2000 tokens（约1500汉字），优先保留"洞察"而不是原始内容。

## Risks / Trade-offs

**[风险] 用户不持续输入 → 系统没有数据**
→ Mitigation：每天22:10前提醒；连续3天无输入则评分降分

**[风险] 初始评分基于推测，不准**
→ Mitigation：标注"待验证"；实际碎碎念开始后逐步校准

**[风险] 用户可能对抗性反驳评分**
→ Mitigation：评分必须有证据；反驳时记录理由，不强行覆盖

**[风险] 系统复杂度提升，维护成本增加**
→ Mitigation：严格按照目录结构执行，不做额外扩展

## Migration Plan

**Phase 1（本change）：**
- 按 OpenSpec 流程创建 proposal → design → specs → tasks
- specs 里每个 capability 都要有详细的 requirement 和 scenario
- tasks 里列出所有文件填充的具体检查项

**Phase 2（执行tasks）：**
- 按 tasks.md 逐项检查和填充
- 每个文件完成时自我检查：深度够不够？PM语境有没有？

**Phase 3（测试）：**
- 手动模拟一次对话，确认记忆加载正确
- 确认触发器逻辑正确
- 确认评分格式正确
