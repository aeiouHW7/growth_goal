# Memory Retrieval System (记忆检索系统)

每次对话开始时，必须执行这个检索流程。不是可选的，是必须的。

---

## 核心原则

**检索 > 记录**

你不是在建仓库，你是在建一个能在需要时被唤醒的记忆系统。

每次对话开始，你必须先问自己：我需要调取哪些记忆？

---

## 三层检索机制

### 第一层：近期记忆（每次必须加载）

**加载最近 7 天的所有碎碎念**

时间范围：当前日期往前推7天

加载内容：
- 所有碎碎念的原始内容
- 每条的"立即洞察"（我自己写的，不是用户的）
- 每条的"维度标签"

输出示例：
```
## 近7天记忆摘要

[04-18] 提到：老板说他"太稳了"，他理解为批评，但行为上没有任何改变
→ 洞察：他可能把"稳"当成保护伞，实际上是不敢冒险
→ 维度：执行、决策

[04-20] 提到：项目又延期了，理由是"需求变了"
→ 洞察：第三次用"需求变了"作为理由，这是模式
→ 维度：风险管理、执行

未解决的问题：
- 他说"稳"但实际上是不敢决策——这个点还没跟他摊开说
- "需求变了"是借口还是真实情况？需要追问
```

**为什么是7天？**
太短看不到模式，太长会让上下文爆炸。7天刚好能抓到重复信号。

---

### 第二层：活跃追踪（按问题标记检索）

**加载标记了"待追踪"的问题**

任何一条碎碎念处理时，如果我觉得"这个点要继续追问"，就在那条记录末尾加上：

```
追踪标签：[待追踪] #维度 #维度
示例：[待追踪] #决策 #执行
```

检索时，加载：
- 所有带 `[待追踪]` 标记的条目
- 排序：最久的在最前面（防止遗忘重要但不紧急的问题）

输出示例：
```
## 待追踪问题

[04-10] "我太稳了" — 还没跟他摊开说他的稳其实是回避
[04-15] 决策模式 — 两次说"选A"，但都选了B，什么原因？

```

---

### 第三层：评分重点（按能力短板检索）

**加载最近一次评分中最低的3个维度**

路径：`memory/profiles/capability_scores.json`

加载原因：这3个维度是下一个月最需要关注的地方。任何与这3个维度相关的碎碎念，都需要特别警觉。

输出示例：
```
## 能力短板（最近一次评分）

1. 政治智慧：4分 — 几乎为零
2. 风险管理：5分 — 知道概念但行为上不操作
3. 决策：6分 — 常年拖延，关键时刻会逃避

这3个维度相关的碎碎念必须进入"挑战模式"
```

---

## 检索顺序

每次对话开始，按这个顺序执行：

1. **加载第一层** — 7天记忆摘要
2. **加载第二层** — 待追踪问题
3. **加载第三层** — 能力短板
4. **判断是否触发** — 结合 pattern_triggers.md 判断是否需要挑战模式

把这三层内容作为系统提示的后缀注入。

## 记忆文件命名规范

```
memory/
├── daily_records/
│   └── YYYY-MM-DD.md          # 每天的原始碎碎念 + 洞察
├── weekly_records/
│   └── YYYY-WW.md             # 每周摘要 + 周考记录
├── monthly_reports/
│   └── YYYY-MM.md             # 月度报告 + 评分更新
├── long_term/
│   └── patterns.md            # 跨时间的模式档案（持续追加）
├── profiles/
│   ├── user_profile.json      # 静态画像（只创建一次）
│   └── capability_scores.json # 评分历史（每次更新）
└── retrieval_index.json       # 轻量索引（按维度标记）
```

---

## 检索索引（retrieval_index.json）

每次碎碎念记录后，更新这个索引：

```json
{
  "dimensions": {
    "认知": ["2026-04-18", "2026-04-20"],
    "决策": ["2026-04-10", "2026-04-15", "2026-04-21"],
    "情商": ["2026-04-19"]
  },
  "patterns": {
    "借口_需求变化": ["2026-04-15", "2026-04-18", "2026-04-22"],
    "回避_老板对话": ["2026-04-10", "2026-04-17"]
  },
  "pending_tracking": [
    {"date": "2026-04-10", "issue": "稳=回避？", "dimensions": ["决策", "执行"]},
    {"date": "2026-04-15", "issue": "决策逃避模式", "dimensions": ["决策"]}
  ]
}
```

这个索引让检索变快。不用读所有文件，根据索引直接知道去哪找。

## 模式档案（long_term/patterns.md）

跨时间发现的模式，永久写入这个文件。

格式：
```
# 模式档案

## [模式名称]
发现时间：
最后一次出现：
出现次数：
涉及维度：
具体表现：
可能原因：
处理方式：
```

示例：
```
## 借口模式：需求变化
发现时间：2026-04-15
最后一次出现：2026-04-22（7天内出现3次）
涉及维度：风险管理、执行
具体表现：每次项目延期都说是"需求变了"，但从不说是谁的需求变了、为什么变
可能原因：不想承认是自己判断失误，或不想得罪提需求的人
处理方式：下次他再这么说，直接问"谁的什么需求变了？为什么变？"
```

## 检索完整性检查

每次对话开始时，自检：
- [ ] 近7天记忆加载了？
- [ ] 待追踪问题加载了？
- [ ] 能力短板加载了？
- [ ] 这三者都注入了上下文？

如果任何一个没做，这条对话就是失忆的，评分结果和模式识别都可能出错。

---

## 内容压缩规则

如果7天记忆太长（超过5000字），需要压缩：

1. 每条碎碎念压缩到一句话摘要 + 洞察
2. 同一维度的合并
3. 重复出现的只保留最新一条（但标记出现次数）

压缩后仍然要有足够的上下文让你做出准确判断。不能因为压缩丢失关键信号。
