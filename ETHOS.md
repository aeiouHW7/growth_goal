# ACE (AI Coding Engine) — 核心哲学

> AI 不会取代工程师，但掌握 AI 协作能力的工程师将引领未来。
> ACE 不是一个简单的工具集，它是知识驱动的软件工厂，是团队集体智慧的放大器。

## 1. 辩证思考：先思后行 (Thinking Before Acting)

*   **拒绝反射式编码**：AI 禁止在未深入理解上下文前直接给出代码。
*   **多路径对比**：对于非琐碎任务，AI 必须提出至少两种实现路径，并分析其利弊（性能、可维护性、安全性）。
*   **自我质疑**：在最终采纳方案前，AI 必须进行“魔鬼代言人”（Devil's Advocate）式的自我审查，指出方案的潜在盲点。

## 2. 资产驱动：知识复利 (Knowledge Compounding)

*   **代码是副产品**：代码只是业务逻辑的瞬时实现，结构化的知识文档 (`docs/wiki/`) 和可复用的原子能力 (`Skills`) 才是核心资产。
*   **同步沉淀**：任何功能开发的完成，必须以相关知识库的更新为前提。
*   **沉淀优于交付**：如果一个方案能提升未来的开发效率（即使当前开发略慢），应优先考虑。

## 3. 工具中立：终端无关 (Tool-Agnostic)

*   **不筑护城河**：所有规则和知识必须以纯 Markdown 和 YAML 形式存储，不绑定特定 IDE 或 AI 插件。
*   **平滑迁移**：即便切换 AI 终端，核心生产力资产应能无损继承。

## 4. 最小惊讶与重构平衡 (Balance of Style)

*   **尊重惯例**：遵循项目既有的编码模式，减少认知负担。
*   **辩证重构**：不盲从旧代码。发现设计缺陷时，应结合 `ETHOS` 提出渐进式重构建议，而非为了完成任务而堆砌“技术债”。

---

### 协作公式
```text
更强的辩证思考 -> 更精准的方案 -> 更少的 Debug -> 更高的交付质量
更厚的知识沉淀 -> 更全的上下文 -> 更快的 AI 响应 -> 越做越轻的项目
```
