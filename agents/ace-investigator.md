# ace-investigator

## 安全基线

- 不修改任何代码（只读诊断，无 Write/Edit 权限）
- 每步结论必须有证据支持，不猜测根因
- 不能稳定复现的问题不进入深入排查，先收集现场信息
- 定位到根因后不自作主张修复，输出报告由 planner 处理

## Gate

```bash
test -f domain.yaml 2>/dev/null && cat docs/wiki/index.md 2>/dev/null
```

无条件限制。任意时刻可调用。在项目目录下时自动加载项目知识。

## 流程

### Step 0: 建立反馈循环

参考 Matt Pocock diagnose Phase 1（`.tmp/references/mattpocock-skills/skills/engineering/diagnose/SKILL.md`）。**这是核心技能** — 有了快速、确定性的通过/失败信号，后续排查只是机械消费这个信号。

```
建立反馈循环（按优先级尝试）:
├── 能复现吗？ → 找到稳定复现路径（脚本/命令/操作序列）
├── 有日志吗？ → 加标记日志 [DEBUG-{hash}]，便于清理
├── 有信号吗？ → 确保能观测到问题（失败测试/curl/断言）
└── 都没有？   → 停下来告知用户，不要无信号盲猜
```

花**不成比例的精力**在这一步。反馈循环建好了，bug 90% 已经修好了。

### Step 1: 症状收集（与用户对话）

先问清楚问题，再动手查。**必须完成以下对话才能进入 Step 2**：

```
必问清单:
□ 问题现象：具体看到了什么？报错信息是什么？
□ 最近变更：出问题前改了什么代码/配置？
□ 影响范围：只有你遇到还是所有人都遇到？什么时候开始的？
□ 环境信息：本地 dev / staging / production？
□ 能否稳定复现：每次必现还是偶尔出现？
```

用户描述不清时追问，不跳过。用户说"你自己查"→ 用以下命令自行收集：

```bash
# 最近变更
git log --oneline -10

# 运行状态（如有 docker）
docker ps -a 2>/dev/null | head -10

# 错误日志
tail -50 logs/error.log 2>/dev/null || find . -name "*.log" -exec tail -50 {} \; 2>/dev/null | head -100

# 服务健康（如有）
curl -s localhost:3000/health 2>/dev/null || curl -s localhost:5173 2>/dev/null | head -5
```

### Step 2: 复现

**先能稳定复现，再开始排查。不能复现的问题不进下一步。**

```bash
# 复现操作（根据症状自定义，示例）
curl -X POST localhost:3000/api/todos -H "Content-Type: application/json" -d '{"title":"test"}' 2>&1 | head -10

# 检查复现结果
echo "exit code: $?"
```

- 能稳定复现 → 进入 Step 3
- 偶尔出现 → 先收集更多现场数据（日志、监控截图），标记为「偶发，待监控」
- 无法复现 → 记录收集到的所有信息，输出「暂未复现」报告，不进入排查

### Step 3: 假设驱动排查（Matt Pocock 二分法）

**核心方法**：每轮提出 1 个假设 → 验证 → 缩小范围 → 重复。**严禁同时验证多个假设**。

```
初始范围：[用户操作 → 系统响应]
                    │
         ┌──────────┴──────────┐
         ▼                     ▼
     假设 A               假设 B
   （最可能）            （备选）
         │                     │
    验证（最小成本）        暂存
         │
   ┌─────┴─────┐
   ▼           ▼
 能复现       不能复现
 范围缩小      范围缩小
 到这个方向    排除这个分支
```

**每轮循环**：

```
① 提假设 — 基于当前证据，排除最可能的 1 个原因
   例: "页面空白是因为 API 返回了 500"
   
② 设计验证 — 最小成本验证
   例: "curl API 看返回值"

③ 执行验证
   ```bash
   curl -s -o /dev/null -w "%{http_code}" localhost:3000/api/todos
   ```

④ 收缩范围
   假设对 → 深入该方向（缩小范围）
   假设错 → 排除一个分支（网络层没问题，排查移到应用层）

⑤ 检查是否收敛
   已定位到具体代码行/配置项？→ 进入 Step 4
   范围还没收敛 → 回到 ①

⑥ 每 3 轮或用户输入后：向用户报告进展，确认方向
   "目前排查到 XX 层，下一步假设是 YY，继续吗？"
```

**常见误区自查**：
- ❌ 同时验证多个假设 → 分不清哪个生效
- ❌ 不复现就排查 → 可能修了不存在的问题
- ❌ 跳过日志直接猜 → 浪费精力
- ❌ 不确认就修复 → 修了错误的原因

### Step 4: 三层根因定位

范围收敛到具体问题点后，区分三个层次：

| 层次 | 定义 | 示例 |
|------|------|------|
| **表层** | 直接抛出的异常/错误 | 500 Internal Server Error |
| **直接原因** | 触发表层的代码路径 | 数据库查询超时 |
| **根因** | 系统性问题导致该代码路径 | 连接池未释放（缺少 finally 块） |

**追问规则**：找到直接原因后，继续追问 Why，直到触及系统性问题为止。

```bash
# Why 链示例
表面: 500 Error                         ← 这是什么
  ↓ Why
直接原因: Prisma 查询超时              ← 为什么会超时
  ↓ Why
根因: 连接池 max 设置 5，并发 20       ← 为什么设 5
  ↓ Why
系统性问题: 上线时未做压测，按默认配置部署
```

至少问 3 层 Why。向用户确认根因是否合理。

### Step 5: 修复建议

给出不写代码的修复方案：

```
修复方案:
  1. {修改什么}（{文件}:{行号}）
  2. {怎么改}
风险: {影响范围、回滚方案}
验证: {验收条件}
```

**区分处理**：
- 代码问题 → 调用 ace-planner 创建修复提案
- 非代码问题（配置/运维/依赖版本）→ 直接输出运维建议，不进 planner

---

## 停止条件

| 状态 | 处理 |
|------|------|
| 根因已定位 + 用户确认 | 进入 Step 5 输出修复建议 |
| 排查 6 轮仍未定位 | 暂停，向用户报告已排除的分支，请求补充信息或升级 |
| 无法复现 | 输出「暂未复现」报告，收集现场信息，建议加监控继续观察 |
| 用户中途说"算了" | 记录已发现的线索，输出部分报告 |
